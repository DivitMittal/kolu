Detecting AskUserQuestion by scraping the xterm buffer

A spike for #905. The issue's fix is a Claude Code PreToolUse hook + sidecar — which means editing the user's ~/.claude/settings.json on every machine. This asks the alternative the title poses: light the "Awaiting input" badge by reading the rendered terminal, so the user configures nothing. Fragility, measured below, is small and test-catchable. Reviewed by /hickey + /lowy (Opus, 2-round debate to consensus) and a Löwy electricity-analogy pass; their consensus edits are folded in and recorded in §9. seedling #905

The one-paragraph version, in Löwy's terms. The consumer-facing receptacle already exists: AgentInfo.state === "awaiting_user", which the badge, dock pip, and OS alert already plug into. Behind that receptacle sit interchangeable power sources for the one fact "the agent is awaiting the user": the live JSONL classifier (starved during the SDK's buffering window), this xterm scrape, and #905's hook. So the work isn't new wiring for consumers — it's adding one more generator behind a socket that's already standardized, and the volatile part ("what a prompt looks like on screen") gets encapsulated in its own utility package. The badge never learns which generator powered it.

1 · Why JSONL polling is blind here

kolu derives every agent's state server-side by tailing the Claude transcript JSONL and classifying the newest message session-watcher.ts:224–274 → core.ts:307 deriveState. The classifier that produces awaiting_user is live, not dead — it already knows these two tools mean "needs the human":

// packages/integrations/claude-code/src/core.ts:231
const AWAITING_USER_TOOLS = new Set(["AskUserQuestion", "ExitPlanMode"]);
// :273-285 toolUseOrAwaitingUser() → classifyByAwaiting() → awaiting_user

The catch, per #905: the Claude Agent SDK buffers the in-flight assistant message (carrying the tool_use block) in memory and flushes it to disk only after the user answers. During the wait the block isn't on disk, so the classifier is input-starved — deriveState reads the prior end_turn entry and reports waiting. schemas.ts:36–43 documents it: "…this case never fires under the current SDK. Fix tracked in #905."

So it's a starved source, not absent machinery — and that has a consequence the first draft missed. The destination (badge chain) is wired-but-unreached; the source classifier is live-but-starved. Once the scrape ships there are two live signals for one fact (and three with the hook), so the plan must state a composition rule — see §6. The receptacle that both lenses confirmed is one enum value the whole attention chain keys off schemas.ts:54 · lifecycle.ts:16-17 classifyByAwaiting · AgentIndicator.tsx:26 · agentState.ts:23.

2 · The "xterm contents" already live server-side

kolu's PTY host keeps a headless terminal emulator per PTY, parses every byte through it, and exposes its contents — first-class RPCs ptyHostSurface.ts:214–225, re-surfaced by the server router as terminal.screenText / screenState router.ts:150–158:

// packages/pty-host/src/ptyHost.ts — :4 "…an @xterm/headless screen mirror…"
// :483  PTY data → headless.write(data, () => publish(data))  // buffer fresh before fan-out
getScreenText(id, startLine?, endLine?): string  // PLAIN TEXT, already VT-parsed — ANSI stripping is free
  • Plain text out — the emulator already applied SGR/ANSI and resolved the grid; we never parse escape codes ourselves.
  • Always live, every PTY — independent of focus or mount state. TerminalHandle.getScreenText already abstracts local vs remote/SSH (#951) local.ts:231, so detection inherits that coverage for free.

Server-side, decisively. Reading this mirror covers background tiles, mobile (client xterm unmounted), and remote/SSH PTYs — none of which a client-side buffer read can promise. Client-side scraping is off the table; it buys nothing and loses coverage.

Three findings that shape the detector:

  • ExitPlanMode is rock-solid — three exact literals, byte-identical across 82 daily releases; as stable as a tool name.
  • AskUserQuestion has thinner fixed chrome — its question + option labels are model-supplied, so detection leans on structure (the figures caret/markers + select footer). The figures lib ships an automatic ASCII fallback (❯→>), so glyph drift is a known finite mapping.
  • No box-drawing glyphs anywhere — Claude's prompt boxes use no ╭/┌ corners. The signature must not depend on borders.

Net: fragility is real but bounded and observable. A break is a discrete event (a specific literal changes), caught by a fixture test on the next Claude bump — not the "constant churn" the generic worry implies (zero churn in 82 releases).

4 · What dodging the hook buys — which generators are connected, not how the current is consumed

The hook isn't free: two stanzas in ~/.claude/settings.json (PreToolUse + PostToolUse), per machine and user, plus an unresolved install-UX decision the issue itself flags. The scrape writes zero files and reads a stream kolu already owns. Coverage the hook structurally can't match without N× setup:

The electricity correction (§9). Scrape and hook are interchangeable generators feeding one receptacle, not a "default" and an "upgrade." "Scrape is the zero-install default, hook adds robustness + clean question text" is a true statement about which generators are connected in a given environment — a deployment fact, never an architectural ranking. How the current is consumed (the awaiting_user badge) never varies; which sources are wired in does. Stating it this way keeps a future Codex/OpenCode scraper or a remote-only hook from re-litigating priority.

5 · The package: packages/xterm-scrape — encapsulate the one volatile thing

The fragile knowledge ("what an interactive terminal prompt looks like on screen") is exactly one volatility, so it lives behind one module boundary (Parnas/Löwy: encapsulate what changes). This is the "even though powering the house is functional, power is not specific to the house's domain" move — the package is a domain-agnostic utility, with no claude import; the Claude integration is merely its first consumer. A dedicated leaf package keeps the blast radius of TUI drift to one place and makes the fixture corpus first-class.

packages/xterm-scrape/                  // pure, browser-safe, zero node:* deps, zero agent imports
  src/signatures.ts   // PRIVATE: declarative per-tool signatures (predicates, not one regex)
  src/detect.ts       // the ONLY export: detectPrompt() + its return type
  src/detect.test.ts  // positives from every fixture + adversarial negatives
  fixtures/           // real getScreenText() dumps, version-stamped — the regression corpus
    exit-plan-mode.2.1.159.txt   ask-user-question.2.1.159.txt
    neg-scrollback-proceed.txt   neg-numbered-list.txt   neg-narrow-wrap.txt

// detectPrompt: a STABLE loop over VARIABLE per-tool signatures (no per-tool branch in the loop)
// Return is a tool-keyed discriminated union — question present iff AskUserQuestion (type-level invariant):
detectPrompt(text): { tool: "ExitPlanMode" } | { tool: "AskUserQuestion", question: string | null } | null

Boolean conjunction, not a confidence score (hickey H5)

Every load-bearing feature is a discrete, byte-stable token — a finite set, not a noisy signal. So detection is an explicit boolean conjunction over the bottom-region text, not a real-valued score against a tunable threshold (which would manufacture a magic number and threshold-drift as a new failure mode):

  • Region gate — only the last N rows near the cursor (getScreenText(id, startLine, endLine)); the live prompt is always at the bottom, history isn't.
  • AskUserQuestion fires iff caret(❯/>) AND option/marker column AND select-footer literal. One feature alone never fires.
  • ExitPlanMode is its own exact-literal predicate (boolean-certain), expressed as a sibling signature in signatures.ts — not a "fast-path" exception wedged inside the AskUserQuestion scorer (hickey H4). The two recognition problems never share a code path; the fixture corpus splits per tool.
  • Add a real-valued score later only if adversarial negatives force it, with fixtures justifying it.

Why the package makes fragility tolerable. Every supported Claude version drops a captured fixture; the test asserts each is detected and each negative rejected. When a future bump changes a literal, CI goes red against the new fixture the moment we add it — the silent-failure mode (badge quietly stops) becomes a loud, located test failure in one package.

6 · Wiring it in — at the providers seam, not in the watcher

This section changed the most under review. The override does not go in the Claude session-watcher.

  • Keep session-watcher.ts JSONL-only (hickey/lowy H1). Today it has zero pty-host imports — pure fs.watch + deriveState; running_background promotion is pure JSONL core.ts:383-385 and is not a precedent for a screen scrape. Injecting getScreenText there would braid JSONL-format volatility with TUI-rendering volatility (two independent change-clocks). Do the work one level up, at startAgentProvider's createWatcher/setAgentMetadataVia callback providers.ts:454-461, which already holds terminalId and the pty-host client and funnels every emitted ClaudeCodeInfo.
  • No pid→PtyId bridge (hickey H2 / lowy M1). TerminalId == PtyId — the same UUID is passed to spawn({id}) and getScreenText({id}) local.ts:361-362, 231-232. There is no namespace to bridge; the screen is reachable directly by the terminalId already in scope, captured as a one-line getScreenText thunk. (Dropped: the old "session↔PtyId seam" decision and its risk row.)
  • Leave the shared AgentProvider.createWatcher contract UNCHANGED (lowy, conceded). Do not thread a screen handle through it anyagent agent-provider.ts:97 — that leaks pty-host (a lower layer) into the agent-agnostic contract and forces Codex/OpenCode to carry/stub an unused concept. Resolve the screen capability at providers.ts and hand it only to the Claude path.
  • One fact, stateless merge (hickey/lowy H3). At the seam, a pure transform on each emitted ClaudeCodeInfo: awaiting_user fires if any contributor says so; the scrape only promotes waiting/end_turn → awaiting_user (never demotes), filling the gap while the JSONL classifier is starved. The contributors are concordant by construction (same tool set), so they reinforce. Write the precedence/merge rule once as a named, commented function; add a "message-flushes-mid-wait" fixture asserting the JSONL and scrape paths agree.
  • Recompute-don't-latch is structural, not a policy (hickey L1). Because the merge is a stateless transform, "no box detected → input passes through unchanged" — there is no scrapeOverride variable to clear, no revert step.
  • One lifecycle object owns the clock (hickey L2). The JSONL fs.watch is silent during the wait, so a poller samples getScreenText at ≈0.5–1s — but the idle-gate + clock + teardown + merge are one object owned by the seam wrapper: started when state goes idle, stopped the instant it goes non-idle or the watcher tears down (tied to watcher.destroy as the single stop point).
JSONL fs.watch ──▶ deriveState(lines) ──▶ ClaudeCodeInfo (BASE — event-driven, no screen, no clock) │ providers.ts seam (startAgentProvider callback, terminalId in scope) │ state idle? lifecycle obj polls getScreenText(terminalId,-N,-1) ▼ mergeAwaitingInput(info, xterm-scrape.detectPrompt(text)) ← named rule, stateless │ promote waiting → awaiting_user iff a prompt is on screen ▼ setAgentMetadataVia ──▶ awaiting_user receptacle ──▶ badge

The receptacle is awaiting_user; the generators (JSONL base + scrape promoter, later the #905 hook promoter) plug in behind the named merge rule. A hook becomes a swap, not a rewrite — the electricity payoff.

7 · Phase 0 — empirical capture done (static)

Captured from the on-disk binaries (results in §3): the exact ExitPlanMode literals, the figures caret + ASCII fallback, the absence of box glyphs, and byte-stability across 82 releases. One thing static analysis can't settle — confirm with a live capture before Phase 1:

Live confirmation (≈1 hr): run a real Claude session in a kolu PTY, trigger ExitPlanMode (and an AskUserQuestion if offered), and dump terminal.screenText. Verify: (a) the prompt lands in getScreenText; (b) main buffer vs alternate screen (?1049h exists in the binary; buffer.active tracks whichever is active, and alt-screen would remove scrollback false-positives); (c) row position relative to the cursor (sets the region gate's N). The dumps become the first fixtures/.

8 · Plan

  1. Phase 0 live capture → first fixtures.
  2. packages/xterm-scrape — private signatures.ts + detectPrompt (boolean conjunction, per-tool signatures, discriminated-union return) + fixture/negative corpus. Pure, no agent imports; ship behind a green suite first.
  3. Server wiring at the providers seam — one lifecycle object (idle-gate + poll clock + teardown), the named stateless mergeAwaitingInput rule, getScreenText thunk closed over terminalId. session-watcher.ts + the shared createWatcher contract stay untouched. Badge lights, self-clears.
  4. Tooltip enrichment (optional) — surface the parsed question (the union already carries it); the hook does this cleaner, so it can come via the hook path later.
  5. Scope decision (state it explicitly): Codex/OpenCode TUI detection is not in this first PR. That collapses the one open framing crux (§9) — under both lenses the first-cut output is identical (per-tool declarative signatures, uniform detectPrompt loop).

9 · What /hickey + /lowy + the electricity pass changed

11 · Recommendation

Build it — server-side, in packages/xterm-scrape, fixture-tested, merged at the providers seam — as the zero-config default. Both review lenses converged on sound-with-changes; the changes are folded in above.

  1. Fragility is measured and small: 0 changes across 82 daily releases; the one durable weak spot (AskUserQuestion's dynamic text) is a boolean conjunction over discrete tokens, and any break is a loud, localized test failure.
  2. The no-config win is the point: nothing written to the user's home; works on remote/SSH/fresh/shared hosts where the hook needs N× setup or silently misses.
  3. One receptacle, swappable generators: the JSONL base + scrape promoter compose by a named stateless merge, so #905's hook drops in later as a swap, not a rewrite — and consumers never learn which generator powered the badge.
  4. Next: the Phase-0 live capture (≈1 hr) to lock the buffer behavior and seed the corpus, then drive the package with /do.
