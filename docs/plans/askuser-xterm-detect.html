Detecting AskUserQuestion by scraping the xterm buffer

A spike for #905. The issue's fix is a Claude Code PreToolUse hook + sidecar file — which means editing the user's ~/.claude/settings.json on every machine. This asks the alternative the title poses: light the "Awaiting input" badge by reading the rendered terminal, so the user configures nothing. The fragility worry is real but — measured below — small and test-catchable. Plan: a dedicated, fixture-tested packages/xterm-scrape, read server-side. seedling #905

The one-paragraph version. Two things you'd assume need building already exist. (1) The destination: awaiting_user is a fully-wired state — enum member, "Awaiting input" label, alert-colored badge, dock attention bucket — that is dead code today, documented in schemas.ts as "never fires under the current SDK." (2) The source: kolu already runs a headless xterm mirror of every PTY server-side and exposes its plain-text screen via an RPC. So "scrape the xterm" is reading a buffer we already maintain and feeding a badge we already shipped. The remaining questions were "how fragile is the signature, really?" and "is dodging the hook's config worth it?" — both now answered with numbers.

1 · Why JSONL polling is blind here (recap)

kolu derives every agent's state server-side by tailing the Claude transcript JSONL and classifying the newest message session-watcher.ts:224–274 → core.ts:307 deriveState. It already knows these two tools mean "needs the human":

// packages/integrations/claude-code/src/core.ts:231
const AWAITING_USER_TOOLS = new Set(["AskUserQuestion", "ExitPlanMode"]);
// :273 toolUseOrAwaitingUser() → classifies the assistant tool_use as awaiting_user

The catch, per #905: the Claude Agent SDK buffers the in-flight assistant message (carrying the tool_use block) in memory and flushes it to disk only after the user answers. During the wait, the block isn't on disk, so deriveState reads the prior end_turn entry and reports waiting, never awaiting_user. schemas.ts:36–43 documents it verbatim: "…this case never fires under the current SDK. Fix tracked in #905."

The bug is a missing signal, not missing machinery. Anything that flips ClaudeCodeInfo.state to awaiting_user during the wait — a hook, or a screen scrape — lights the badge with zero downstream work. The whole "attention" chain (label, color, dock pip, OS badge, alert) already keys off that one enum value schemas.ts:50–56 · agentDisplay.ts stateLabels · AgentIndicator.tsx:26 · agentState.ts:23.

2 · The "xterm contents" already live server-side

kolu's PTY host keeps a headless terminal emulator per PTY, parses every byte through it, and exposes its contents two ways — both first-class RPCs ptyHostSurface.ts:214–225, already re-surfaced by the server router as terminal.screenText / screenState router.ts:150–158:

// packages/pty-host/src/ptyHost.ts — :4 "…an @xterm/headless screen mirror…"
// :483  PTY data → headless.write(data, () => publish(data))  // buffer fresh before fan-out
getScreenText(id, startLine?, endLine?): string  // PLAIN TEXT, already VT-parsed — ANSI stripping is free
  • Plain text out — the emulator already applied SGR/ANSI and resolved the grid. We never parse escape codes ourselves.
  • Always live, every PTY — independent of focus or mount state, and it's the established home for stream-derived signals (the same headless terminal already runs OSC handlers for cwd / title / shell-integration ptyHost.ts:415,433,446).

Server-side, decisively. Reading this mirror covers background tiles, mobile (where the client xterm is unmounted), and remote/SSH PTYs — none of which a client-side buffer read can promise. Client-side scraping is therefore off the table here; it buys nothing and loses coverage. (Footnote: the client buffer + SerializeAddon exist via terminalRefs.ts if ever needed, but there's no reason to.)

3 · Quantifying the fragility — "TUI pixels" measured, not feared

The honest worry was "we're scraping a closed, fast-moving TUI's output." I measured it against four Claude Code builds on disk (2.1.77, 2.1.81, 2.1.92, current 2.1.159) by extracting the bytes the binary actually emits.

The prompts are anchored on stable literals and a stable library — not arbitrary pixels. What's actually on screen, and how durable each part is:

ElementOn-screen tokenSource2.1.77→159

Two findings that directly shape the detector:

  • ExitPlanMode is rock-solid — three exact literals, byte-identical across 82 daily releases. This one is essentially as stable as a tool name.
  • AskUserQuestion has thinner fixed chrome — its question text and option labels are model-supplied, so detection leans on structure: the figures caret/markers + the select footer, not fixed phrasing. The figures library ships an automatic ASCII fallback (❯→>), so glyph drift is a known finite mapping, not open-ended.
  • No box-drawing glyphs anywhere — Claude's prompt boxes use no ╭/┌ corners. The signature must not depend on borders. (A naïve "look for a framed box" detector would have failed.)

Net: fragility is real but bounded and observable. A break would be a discrete event (a specific literal changes), caught by a fixture test on the next Claude bump — exactly the "solid test coverage" posture, made concrete in §5. It is not the "constant churn" the generic worry implies: zero churn in 82 releases.

4 · What dodging the hook actually buys — the no-config benefit, quantified

The hook isn't free. It means installing two stanzas into ~/.claude/settings.json (PreToolUse + PostToolUse), per machine and per user, plus an unresolved install-UX decision the issue itself flags (auto-edit the user's settings vs. an opt-in kolu claude install-hooks). The scrape writes zero files and reads a stream kolu already owns. Concretely, the scrape covers cases the hook structurally cannot — or can only with N× the setup:

ContextHookServer-side scrape
Local Claude, hook installedworks (robust)works
Remote / SSH PTY (#951)needs install on every remote boxworks (kolu proxies the PTY)

So they're complementary, and the scrape is the better default: zero-install, works everywhere kolu can see a PTY, and agent-agnostic (the same package can recognize Codex/OpenCode TUI prompts). The hook stays available as a robustness upgrade for local Claude and for clean question text — but it should be opt-in, not a prerequisite for the badge to work at all.

5 · The package: packages/xterm-scrape — encapsulate the one volatile thing, test it hard

The fragile knowledge ("what an interactive terminal prompt looks like on screen") is exactly one volatility — so it belongs behind one module boundary (Parnas / lowy: encapsulate what changes). A dedicated leaf package keeps the blast radius of any future TUI drift to one place, and makes the test corpus first-class.

packages/xterm-scrape/                  // pure, browser-safe, zero node:* deps (mirrors schemas.ts discipline)
  src/signatures.ts   // declarative per-tool signatures: feature predicates, not one regex
  src/detect.ts       // detectPrompt(screenText, opts) → { tool, confidence, question? } | null
  src/detect.test.ts  // positives from every fixture + adversarial negatives
  fixtures/           // real getScreenText() dumps, version-stamped — the regression corpus
    exit-plan-mode.2.1.159.txt
    ask-user-question.2.1.159.txt
    neg-scrollback-proceed.txt   neg-numbered-list.txt   neg-narrow-wrap.txt

Heuristics, not a substring match (the false-positive answer)

buffer.active includes scrollback, so a lone "proceed" or a numbered list in agent output must not trip the badge. detectPrompt scores the bottom region near the cursor (via getScreenText(id, startLine, endLine)) on multiple co-occurring features and fires only past a threshold:

  • Region gate — only the last N rows; the live prompt is always at the bottom, history isn't.
  • Multi-feature score — caret ❯/> and an option/marker column and (ExitPlanMode literal or select-footer hint). One feature alone never fires.
  • ExitPlanMode fast-path — its three exact literals are high-confidence on their own.
  • Recompute, don't latch — re-scored each poll; absence of the box ⇒ revert to the JSONL-derived state. No separate "dismiss" detector.

Why the package makes the fragility tolerable. Every supported Claude version drops a captured fixture into fixtures/; the test asserts each is detected and each negative is rejected. When a future bump does change a literal, CI goes red against the new fixture the moment we add it — the silent-failure mode (badge quietly stops working) becomes a loud, located test failure in one package. That is the entire bet, made testable.

6 · Wiring it in (server-side)

Two integration facts beyond the package:

  • session ↔ PtyId. The claude SessionWatcher is keyed by Claude session and imports nothing from pty-host today. The bridge belongs in the server layer that already owns both — LocalTerminalBackend wires the pty-host client and matches sessions by foreground pid terminalBackend/local.ts · agent-provider.ts resolveSession(foregroundPid). Expose that pid→PtyId association to the detector.
  • Its own clock. The JSONL fs.watch is silent during the wait, so the scrape needs a timer: poll getScreenText at ≈0.5–1s, gated to run only while JSONL state is idle (waiting/end_turn). When detectPrompt fires, promote to awaiting_user through the existing onUpdate/claudeInfoEqual gate — the same spot running_background is promoted today core.ts:378–385; session-watcher.ts:261.
deriveState(lines) ─ idle? ─▶ poll getScreenText(ptyId, -N, -1) │ │ (JSONL path) xterm-scrape: detectPrompt(text) │ │ tool? confidence ≥ θ ▼ ▼ state = waiting ───override──▶ state = awaiting_user ──▶ onUpdate ──▶ badge (box gone ⇒ revert to JSONL state)

7 · Phase 0 — empirical capture done (static)

Captured from the on-disk binaries (results are §3). Settled: the exact ExitPlanMode literals, the figures caret + ASCII fallback, the absence of box glyphs, and byte-stability across 82 releases. One thing static analysis can't fully settle — confirm with a live capture before Phase 1:

Live confirmation needed (≈1 hr): run a real Claude session in a kolu PTY, trigger ExitPlanMode (and an AskUserQuestion if the model offers one), and dump terminal.screenText. Verify: (a) the prompt text actually lands in getScreenText; (b) whether it renders in the main buffer or the alternate screen — ?1049h sequences exist in the binary, and buffer.active tracks whichever screen is active, so either is fine, but alt-screen would remove scrollback false-positives entirely; (c) its row position relative to the cursor (sets the region gate's N). The dumps become the first fixtures/.

8 · Plan

  1. Phase 0 live capture → first fixtures (above).
  2. packages/xterm-scrape — signatures.ts + detect.ts + the fixture/negative test corpus. Pure, no node deps; ship behind a green test suite first.
  3. Server wiring — pid→PtyId seam + idle-gated poller in the claude integration; promote to awaiting_user via the existing gate. Badge lights, self-clears.
  4. Tooltip enrichment (optional) — surface the parsed question on the indicator; this is where the hook is cleaner, so it can come later or via the hook path.

10 · Recommendation

Build it — server-side, in packages/xterm-scrape, fixture-tested — as the zero-config default.

  1. The fragility is measured and small: 0 changes to the prompt signature across 82 daily releases; the one durable weak spot (AskUserQuestion's dynamic text) is handled by structural heuristics, and any future break is a loud, localized test failure — not a silent one.
  2. The no-config win is the point: nothing written to the user's home, and it works on remote/SSH/fresh/shared hosts where the hook needs N× setup or silently misses.
  3. Keep #905's hook as an opt-in robustness upgrade for local Claude and clean question text — complementary, not a prerequisite.
  4. Next: the Phase-0 live capture (≈1 hr) to lock the buffer behavior and seed the corpus, then drive the package with /do.
