Three volatility axes the design carves out

Lowy's question: what changes for its own reasons, and is each axis isolated to one module? The design carves three:

AxisEncapsulated byReason
Where a terminal's state lives (this machine vs. a specific SSH host)TerminalBackend interface + location discriminatorThe only thing local-vs-remote changes at the kolu-server level. Everything downstream — the router, the surface, the orchestrators — talks to the TerminalBackend and doesn't know which location kind it has.
How TerminalBackend reaches its agent (transport, framing, reconnect)HostSession + @kolu/surface/links/stdioThe transport changes for its own reasons (loopback today + ssh tomorrow, mTLS-over-TCP later). The backend asks HostSession for a typed surface client; doesn't care how it connects.

Hickey's complement: an earlier design draft had per-domain remote providers (RemoteGitInfoProvider, RemoteAgentProvider, RemoteFsProvider, …) — each a transport adapter wearing a domain costume. The single-TerminalBackend cut has zero domain knowledge crossing the transport boundary. The agent runs the unmodified local providers; the boundary just ships their output.

Lessons informing the design

Six lessons from the exploratory prototype that shape the choices below. Each names a specific failure mode someone redoing this would otherwise hit.

#LessonConcrete design choice
1 Hand-rolled transport adapters mask framework work. If the code you're writing in your application would benefit any oRPC consumer (not just yours), it belongs in the framework. R-1.5 landed stdio link, loopback link, peer-server, in-memory channel/cell/publisher in @kolu/surface — generic, zero Kolu specifics — plus the @kolu/surface-nix-host package for the ssh + Nix glue. R-2 consumes them. Don't write a stdio adapter inside packages/server/.
2 Per-entity reactive state wants collections, not parameterized streams. "What changed about entity X" is what Collection<K, V> is for; snapshot+key-tracking semantics come built-in. Stream is for keyed compute (search results); collection is for keyed state. The agent surface has ONE terminalMetadata collection keyed by terminal id, NOT eight parameterized Stream<{id}, T> entries. Aggregation happens on the agent; RemoteTerminalBackend subscribes once.
3 Framework code needs an end-to-end demo, not just unit tests. The demo is a falsifiability test: if writing it against the primitives feels awkward, the primitives are at the wrong altitude. R-1.5 is not "done" until the example app's worker/ sub-demo exercises the stdio link, peer-server, and in-memory channel with snapshot+delta semantics observed via the browser UI.
4 "Shadow entry sync, I/O async" belongs in the TerminalBackend.spawnPty contract. Registering a connecting-state entry synchronously, then doing all I/O on a later tick, is what makes the instant-tile UX work. R-1's TerminalBackend.spawnPty contract documents the sync-shadow / async-tail pattern. LocalTerminalBackend's I/O is instantaneous so it's a no-op there; RemoteTerminalBackend follows the same contract.
5 Wire-shape drift is a real bug class — fail fast, no defaults. If kolu-server and the remote agent are built from different sources, every RPC 404s silently (kolu-server speaks surface.X, agent serves something else). R-2's install.ts requires KOLU_AGENT_FLAKE_REF. No default — throws at first remote-spawn with an error message naming the bug class. Operator chooses the ref explicitly (dev branch / release tag).
6 Cold remote realisation can take minutes — design for it. First nix run on a fresh remote builds the closure; without UX accommodation the user stares at a blank tile. R-2 designs in three mitigations from the start: (a) instant tile + DisconnectedOverlay's connecting-state spinner; (b) kolu-server pre-generates the terminal id so it can register before the spawn RPC roundtrips; (c) heartbeat deferred via session.markConnected() — doesn't start until first RPC succeeds, so the 5s × 5 timeout doesn't fire during the build.

R-1 — TerminalBackend abstraction + LocalTerminalBackend shipped

Landed on master as #981. The description below documents the shape that landed; future phases consume it as-is.

User outcome: none. Pure architectural work — the server spine is rewritten around one interface; behavior is byte-identical.

What "TerminalBackend" means

A TerminalBackend is the per-terminal world a terminal lives in. Concretely: what process owns the PTY, what filesystem the Code-tab reads, where the git watcher runs, where the agent-state providers (Claude Code, OpenCode, Codex) observe their session files. Every per-terminal stream and every per-host one-shot op goes through this object.

Two concrete kinds:

  • LocalTerminalBackend — this kolu process. The PTY is spawned in-process via node-pty; providers watch local files via @parcel/watcher; git/fs ops shell out locally.
  • RemoteTerminalBackend — a specific SSH host. The PTY runs in the kolu --stdio agent on that host; every method proxies via oRPC over the agent's surface (R-2).

What it replaces. Without this interface, the kolu-server's meta/*.ts orchestrators import kolu-git, kolu-anyagent, kolu-github directly and watch local files. Adding remote support without a backend interface means each integration grows a location parameter or a parallel Remote* twin — transport dispatch braided into every domain function. The TerminalBackend interface concentrates the local-vs-remote decision at one seam (terminal creation, via getTerminalBackendFor(location)); every consumer downstream just talks to backend.X and never asks "which kind?"

What dissolves into it. LocalTerminalBackend absorbs the kolu-server's meta/*.ts per-terminal orchestrators (claude-code reconciler, codex reconciler, opencode reconciler, github PR poller, foreground-process detector, git watcher, cwd watcher) as private wiring inside its spawnPty. These files delete as a kolu-server-side layer in R-1 — they survive only as their integration-package source (kolu-claude-code, kolu-git, etc.) that LocalTerminalBackend calls.

Why this is the right cut. See the volatility axes: "where a terminal's state lives" is one independent axis. Carving the system at that axis means every other piece of the design (router, surface, restore, UI) treats local and remote as the same shape. The only place that pattern-matches on location.kind is getTerminalBackendFor, the resolver itself.

The interface

interface TerminalBackend {
  readonly id: TerminalLocation;        // { kind: "local" } | { kind: "ssh", host }
  spawnPty(opts: PtySpawnOpts): Promise<TerminalHandle>;
  terminalChannel<K>(terminalId, kind: K, signal?): AsyncIterable<TerminalChannelMap[K]>;
  killTerminal(terminalId): boolean;
  killTerminalEntry(entry): void;       // bulk-kill path, takes entry directly
  uploadFile(terminalId, name, base64Data): Promise<string>;
  readonly fs: TerminalBackendFs;               // listAll, readFile, subscribeRepoChange, subscribeFileChange
  readonly git: TerminalBackendGit;             // getDiff, getStatus
}

Invariants

  1. Kill convergence. TerminalBackend.killTerminal(id) is the sole termination path. TerminalHandle does NOT carry dispose() — handle-as-control-surface and kill-as-lifecycle are two roles that get smuggled through one method if you let them.
  2. Snapshot-then-delta on every stream. terminalChannel<K> first yield is a full state snapshot; subsequent yields are deltas (the rule in .claude/rules/streaming.md). This is what makes oRPC's ClientRetryPlugin reconnect transparently — and what makes the equivalent on the kolu-server ↔ agent hop also reconnect transparently in R-3.
  3. TerminalBackend owns its filesystem. TerminalBackendFs/TerminalBackendGit cover BOTH one-shot ops AND watcher subscriptions. Same axis ("where the FS lives"), one place to dispatch.
  4. Sync shadow entry, async I/O. spawnPty registers a TerminalProcess entry synchronously (so the tile renders immediately), then performs any I/O on a later tick. LocalTerminalBackend's I/O is instantaneous so this is a no-op there; RemoteTerminalBackend needs minutes for cold nix run realisation and the contract is what makes the instant-tile UX work.

How the refactor stayed safe

  • Atomic refactor — single PR. Six commits on the PR: the primary spine (interface + LocalTerminalBackend + resolver + meta/* deletion), two Hickey/Lowy fixes (drop wrap-closures, unify currentAgent onto the record), three code-police elegance fixes (extract localBackend const, drop redundant pre-check, restore doc comments). No transitional state, no feature flag — anything that survived into master is part of the new shape, not residue.
  • Behavior preserved structurally. LocalTerminalBackend invokes the same integration-package functions (kolu-claude-code, kolu-git, kolu-github, …) the orchestrators called. The publish-routing fence between persisted and live metadata fields (updateServerMetadata fires terminals:dirty; updateServerLiveMetadata doesn't) and the recency-bump rule (shouldBumpRecencyForAgentChange) are covered by the moved unit tests. R-1.5 — @kolu/surface framework gaps shipped

User outcome: none directly. To framework consumers: "@kolu/surface now supports subprocess-worker patterns" — same typed reactive primitives that power browser UI now work for sandboxed compute workers, IDE-extension language servers, remote SSH workers, etc. Plus a new sibling package @kolu/surface-nix-host that turns "ship a typed surface to another machine over ssh + Nix" into a one-liner consumers can call.

Surface was browser-coupled at three spots. R-1.5 closed those gaps with generic additions — zero Kolu specifics, useful to any oRPC consumer over a non-WebSocket link — and extracted the remote-deployment glue into its own package once the demo had proven its shape.

What landed in @kolu/surface

  1. Stdio link adapter (@kolu/surface/links/stdio) — StdioRPCLink + createStdioCellsClient. Wires an oRPC ClientPeer (from @orpc/standard-server-peer) to a Readable/Writable pair via base64+newline framing. Direction-neutral options (read / write) so client and server use the same shape.
  2. In-process loopback (@kolu/surface/links/loopback) — createLoopbackPair() returns crossed PassThrough streams so a server and client in the same process exercise the same framing as a real subprocess pair. Enables symmetric "local backend wrapped in the same client shape as a remote backend" patterns; also useful for testing.
  3. Peer server (@kolu/surface/peer-server) — serveOverStdio({ router, transport }) pumps any typed oRPC router through ServerPeer. Default transport is process.stdin / process.stdout; loopback consumers pass the server half of a LoopbackPair. Defensively redirects console.log to process.stderr to mitigate lesson #4.
  4. In-process pub/sub family — inMemoryChannel<T>(), inMemoryCell<T>(), inMemoryPublisher(), and inMemoryChannelByName() in @kolu/surface/server, next to the existing inMemoryStore<T>. Single-process broadcast pub/sub for surfaces served from a Node-only process where publisherChannel's external publisher dep is overkill. inMemoryChannelByName is the load-bearing piece for implementSurface's keyed channels — without dedupe by name, the framework's publish-site and subscribe-site land on different Channel<T> instances and every delta is lost.

What landed in the new @kolu/surface-nix-host package

The remote-process-monitor demo grew a transport+lifecycle layer that turned out to be wholly generic — ssh subprocess management, .drv provisioning, snapshot-then-delta connection state, exponential-backoff reconnect, ref-counted host sessions. By the end of the PR review (after hickey + lowy structural passes and a code-police elegance pass), it was clear this should live as its own package both for re-use in R-2 and to give reviewers a clean package boundary instead of asking them to re-verify ssh-and-Nix machinery on every future Kolu PR. The split happened in the same PR (commit 1d9fb7f8) once the shape stabilised.

Public API (generic over C extends ContractRouter):

  • HostSession<C> — one ssh subprocess per (host, drvPath, binary). Ref-counted via pin()/acquire()/release(). State machine. onState snapshot-then-delta. currentClient() + isDestroyed() for the consumer's reconnect-loop.
  • provisionAgent({ host, drvPath, onProgress }) — nix copy --derivation + ssh nix-store --realise orchestration. Nix is the contract, not the implementation — no tarball / Docker / prebuilt fallback exists or will.
  • mirrorRemoteCollection<K,V>(opts) — generic per-key bridge: subscribe to the agent's keys stream, open per-key get(key) streams, pump values into the consumer's local collection. The fit for R-2's terminalMetadata (small-N, per-key reactive).
  • waitForNextClient(session, previous) — the reconnect-loop primitive: blocks until the session produces a fresh AgentClient<C> identity-distinct from the previous iteration's. Underwrites "reconnect → state reconciles, no ghosts" without consumers having to write the loop themselves.
  • buildAgentCommand, isLocalHost, forEachLine — helpers exported because R-2 needs them too.

Remote process monitor (remote subprocess over SSH; "miniature Kolu")

New three-tier web app in packages/surface/example/remote-process-monitor/ — browser UI on top, a parent Node server in the middle, the remote agent at the bottom over ssh-stdio. The parent re-exposes the agent's surface to the browser, so the browser is a thin SolidJS client subscribing to a typed surface identical in shape to the one the notes app already uses:

Browser (SolidJS UI)
   │  WebSocket (oRPC — the framework's existing browser transport)
   ▼
Parent server (Node — packages/surface/example/remote-process-monitor/)
   │  ssh stdio (the R-1.5 stdio link — the new transport this PR adds)
   ▼
Remote host: process-monitor-agent (Node, --stdio mode)
   │  /proc/* on linux, sysctl on darwin
   ▼
Kernel

This is the same three-tier shape Kolu R-2 will have (browser ↔ kolu-server over WebSocket ↔ kolu-agent over ssh-stdio) — just with process data instead of terminal data. The parent's bridging role is what R-2's RemoteTerminalBackend ends up performing for terminals.

The parent server spawns the agent by shipping its derivation (not a realised closure) to the remote and realising it there — so a darwin parent can drive a linux remote without ever needing a cross-platform builder. This was the v1→v2 pivot during the PR (the original plan above said "build the closure locally" but that fails cross-arch; KOLU_AGENT_DRV + remote-realise was the resolution):

drv=$(nix eval --raw ".#packages..process-monitor-agent.drvPath")
nix copy --derivation --to ssh-ng://$host $drv
ssh $host nix-store --realise $drv          → /nix/store/...-agent
ssh $host $agentPath/bin/process-monitor-agent --stdio

The operator supplies KOLU_AGENT_DRV (or the just dev recipe computes it via ssh $host uname -ms). The agent reads /proc/* on linux or sysctl on darwin and exposes: a processes collection keyed by PID, a small-N cpuCores collection (4–32 per-core busy%), a system cell (load, mem, uptime), a connection cell (parent-owned lifecycle state), a processesSnapshot stream (bulk discriminated {snapshot, entries[]} \| {delta, upserts[], removes[]} for the 600+ PIDs case where per-key fan-out is N+1-slow), and a kill(pid, signal) procedure. Cold first connect shows copy + realise progress in the UI before dropping into the live view.

Connect-time states the UI must show — the same lifecycle RemoteTerminalBackend will need to surface in R-2:

1. "Copying agent to remote…"      (nix copy in flight)
2. "Connecting…"                   (ssh subprocess spawned, awaiting handshake)
3. "● connected"                   (first RPC roundtripped → heartbeat starts)
4. "✕ disconnected — Reconnect"    (heartbeat missed × 5 → reconnect button)

Primitives exercised: stdio link (over ssh's stdin/stdout), peer-server, snapshot-then-delta semantics across a network hop (keyed-collection snapshot + per-key deltas as processes appear/disappear/change), in-memory channel on the remote side for the per-tick snapshot bus.

User-facing benefit: one-click live top for any host you can ssh to. No remote shell, no terminal multiplexer, no breaking out of the browser. The collection refreshes every 2s; reconnect-after-blip re-snaps the process list and the client reconciles against its last view via the framework's keyed-snapshot invariant.

Why this demo mattered as a falsifiability test. It's Kolu remote terminals in miniature — same transport stack (ssh + nix copy --derivation + nix-store --realise), same cold-realisation wait pattern, same protocol-channel-is-stdout gotcha (lesson #4), and crucially the same shape Kolu's terminalMetadata collection has: a keyed-snapshot collection of structured per-id state with deltas as state evolves. The demo proved R-1.5's primitives can express this cleanly — so R-2 reduces to "Kolu-specific consumer of the same shape" — RemoteTerminalBackend's parent-side wiring is a structural twin of this demo's parent-side wiring, with several primitives (HostSession<C>, provisionAgent, mirrorRemoteCollection, waitForNextClient) already extracted into @kolu/surface-nix-host for R-2 to consume directly.

Footguns we found (now documented + tested)

  • implementSurface(...).router returns a fragment, not a top-level router. Re-wrap with implement(contract).router(fragment) before handing to StandardRPCHandler, or every request 404s. documented + regression-test pinned — see packages/surface/src/implementSurface.test.ts and the "Router wrapping (footgun)" section of packages/surface/README.md. The notes example silently had this bug; demo cold-start surfaced it on the first end-to-end run.
  • Stdio's stdout IS the protocol channel in agent processes. Any pino log line corrupts the next frame and the client peer dies with SyntaxError: Unexpected token '«'. mitigated + reproducer shipped — serveOverStdio defensively redirects console.log to process.stderr in the default-transport case; the agent's --broken-stdout-log flag deliberately reproduces the failure mode for a regression test. R-2 still needs the same redirect at the pino layer.
  • inMemoryPublisher's channel-name dedupe is load-bearing. Without it, the framework's keyed-channel publish-site and subscribe-site land on different Channel<T> instances and every delta is silently dropped. fixed in R-1.5 — inMemoryChannelByName() exists for this case; inMemoryPublisher evicts empty per-name channels on idle to avoid unbounded growth on long sessions (~600 PIDs every poll for the process monitor demo).
  • Stdio link doesn't auto-reconnect mid-stream (the streams die with the agent process). The ClientRetryPlugin alone is not enough — consumers must loop on waitForNextClient(session, previous) to bind to each fresh AgentClient the session produces post-respawn. R-1.5's bridge does this; R-2 inherits the same pattern.

R-1 deliberately dissolved the providers into LocalTerminalBackend as private functions — the right cut for single-host kolu, where co-locating each provider with its lone consumer was simplest. Originally specified as: installAgent(host) probes nix; remoteAgentCommand(host) returns argv. Resolver throws at first remote-spawn if unset, error message names the bug class.
  • Cross-arch: the .drv-ship + remote-realise approach (R-1.5's design) is what enables darwin parent ↔ linux remote and vice versa with no remote-builder config. The kind names match AgentTerminalMetadata field names exactly — direct indexed access, no switch.
  • connectionState is in-process — it's the kolu-server's view of HostSession state, NOT a thing to fetch from the agent. RemoteTerminalBackend.terminalChannel(id, "connectionState") wraps HostSession.onState in an AsyncIterable directly.
  • PTY data and command-run streams stay separate: terminalData, terminalCommandRun, terminalTitle all flow through their own per-id streams (client.terminalData.get({key: id}) etc.). High-throughput, latency-sensitive — keeping them out of the metadata collection avoids backpressure interference. Each gets its own pump in the bridge.

PR shape — one PR, six commits

R-2 lands as a single PR, not a stack. The slices are conceptual breakdowns for review tractability, not deployable units — no slice ships user-visible value on its own, so splitting introduces transitional states on master where remote terminals are half-wired (e.g. session reachable but no metadata). For an in-flight feature with a clean "all or nothing" success criterion (a remote tile actually working end-to-end), the cost of the single big PR is lower than the cost of fanning the agent/server contract changes across multiple round-trips.

  • Commit cadence inside the PR: one commit per slice (2a, 2b, 2c, 2d, 2e, 2f), in order. Each commit must typecheck and pass tests on its own. Reviewers can step through commit-by-commit; the merge is one revertable unit.
  • Why not a stacked PR series: the agent surface contract (2b) is co-evolving with the consumer (2d-2e). Splitting introduces two-way version churn during review. The same reasoning is why R-1.5 shipped as one PR (#984) despite spanning 14+ commits.
  • Feature-flag the new backend off by default? No. getTerminalBackendFor(location) already pattern-matches on location.kind — local terminals continue to resolve to LocalTerminalBackend. The new remote arm is opt-in via the user explicitly choosing a remote host in the UI; no flag needed.
  • Slice-level done criteria still serve as commit-message gates: if 2c's commit doesn't pass its "done when" verification, it doesn't land — review can demand a rebase before continuing.

R-3 — Version handshake + session-ID persistence (network-blip resilience)

User outcome: laptop sleep / wifi drop / suspend-resume leaves remote terminals alive with full scrollback. This is the axis Zed itself doesn't solve — Zed terminals die with the ssh subprocess. Kolu can go further because the reattach machinery already exists as @kolu/pty-host from R-4. R-3 just persists the session identifier and adds the contract-version handshake on top.

Most of what was originally R-3 became R-4. Pre-flip, R-3 owned attachPty(id), stable session IDs, idle-TTL GC, and the snapshot-then-delta semantics on reattach. R-4 now ships all of that inside @kolu/pty-host (the same machinery makes local PTYs survive kolu-server restart). R-3's remaining incremental scope is small:

  • Persist remoteSessionId. Add the field to ServerPersistedTerminalFieldsSchema + thread it through the kolu-server's saved-terminal record. On reconnect, the kolu-server calls session.surfaceClient.terminal.attach({ id: remoteSessionId }); the agent's PtyHost.attach() already yields the snapshot-then-delta stream R-3 wanted. ~30 LoC.
  • Retire the --refresh hack. system.version already exists from R-4 (the long-lived local daemon forced it to ship in R-4 — kolu-server-vs-daemon contract skew detection couldn't wait). R-3 just removes the blanket --refresh on every spawn at the kolu-server call site, since the version handshake now triggers a single explicit refresh + reconnect on skew with a clear error message. ~50 LoC of call-site cleanup, no new agent-side work.
  • Pre-warm on host-picker open: fire-and-forget provisionAgent({ host, drvPath, onProgress: noop }) for the highlighted host so the first real terminal-spawn doesn't wait on cold realisation. ~20 LoC.
  • Close the overflow-recovery loop (re-attach on drop). @kolu/pty-host's bounded Channel sheds a slow subscriber by ending its attach stream — drop-slow is the correct bounded-memory primitive for a remote agent that outruns the socket (one wedged browser tab can't OOM kolu-server). But a clean stream-end is not a transport error, so today the client's consumeStream simply stops: ClientRetryPlugin's onRetry fires on reconnect, not on a clean end. R-3 closes the loop — an overflow-drop must drive a fresh re-attach (snapshot-then-delta), made distinguishable from a PTY-exit close so a dead terminal isn't resurrected. R4a deliberately ships the primitive without this consumer-side recovery (in-process, the 10k-chunk cap is unreachable over a localhost socket); it only becomes load-bearing once a real network sits between agent and kolu-server. ~40 LoC, client-side.

~130 LoC total — down from R-3's original ~470 LoC scope. The work isn't smaller because it was less important; it's smaller because R-4 made the structural primitive the reattach was built on into a first-class published one. R-3 is now mostly schema + handshake + UX polish.

Why keep terminal bytes and metadata on separate streams (vs. one fat stream)? Terminal bytes are high-throughput, latency-sensitive raw streams; metadata is structured, lower-frequency snapshots. Multiplexing them onto one channel tangles backpressure semantics. The agent surface already gives them separate slots (terminalData stream vs. terminalMetadata collection); R-3 doesn't change that. The same is true of PtyHost.attach()'s { snapshot, deltas } shape — bytes stay bytes, metadata stays in the collection.

R-2 prototype (#994) — retro: what we'd carve differently

Retro — what the prototype taught us, and what we'd carve differently

The plan said "one PR, six commits." The prototype landed as one PR, 46 commits — ~7× the plan's cadence. The slice shapes were mostly right; the slice boundaries were wrong in two places and one whole concern was missing from the plan entirely. The lessons below would inform any future agent-style integration (Kolu-on-NixOS hosts, Kolu-on-WSL, MCP-server adapters, sandboxed compute workers).

Three things the plan didn't see coming

SurpriseWhat we shippedWhy the plan missed it
Provider DAG had to be extracted, not re-used in place. New file packages/server/src/terminalBackend/providers.ts (~490 LoC) holding the per-terminal provider DAG (claude-code/codex/opencode detectors, git watcher, GitHub PR watcher, agent-command tracker, foreground-process observer). Both LocalTerminalBackend AND agent.ts consume it via a ProviderHooks indirection so neither knows which host it's running on. The plan said "the agent runs the unmodified local providers." That's the right intent, but the providers were tangled into LocalTerminalBackend as private functions (R-1's deliberate design — see "What dissolves into it"). Reusing them across machines required an explicit extraction step the plan didn't budget for. This is the load-bearing refactor of R-2 and deserved its own slice.
Parent-side screen-state mirror leaked from R-3. RemoteTerminalBackend runs its own @xterm/headless + SerializeAddon instance fed by the agent's data pump, so the kolu-server's terminal.attach can serve scrollback locally on WebSocket-reconnect without round-tripping back to the agent. Requires createRequire shenanigans for CJS interop. The plan punted scrollback-on-reconnect to R-3 ("PTY ownership + reattach"). In reality, the parent-side reconnect (browser ↔ kolu-server WebSocket drop) hits before the R-3 case (kolu-server ↔ agent ssh drop), and snapshot-then-delta semantics demand some snapshot source. Without a parent-side mirror, every browser refresh re-RPCs the agent — both slow and a denial-of-service risk on cold agents.
Connection lifecycle has 5+ states and 4 distinct invariants, not the 3 the plan listed. States: copying → connecting → connected → disconnected → destroyed. Invariants added during 4 rounds of Codex review: (a) forceReconnect distinct from destroy (SIGKILL the child but keep the pool entry); (b) sessionPromise clears on reject so a failed first connect doesn't brick the host; (c) parent-side connectedAcked flag waits for first metadata-collection yield, not just first RPC; (d) agent natural-exit calls publishMetadataRemove through the surface bus so the parent sees the removal as a delta. The plan's heartbeat slice (2c) said "5s × 5 misses → markUnhealthy" and called it done. That's the steady-state rule. The real complexity is the edge transitions: cold-start can't trigger heartbeat (lesson #6, planned), but also can't trigger reconnect storm if the child dies during nix-realise (not planned); a destroyed session in the pool needs to be replaced not reused (not planned); an agent exit-via-SIGTERM is different from exit-via-disconnect (not planned).

How we'd split R-2 differently (now realized)

The prototype's six slices conflated "the shared engine" with "the consumer code" — Codex review kept catching lifecycle holes precisely because the diff didn't isolate the shared boundary. The cleaner carve is the 5-slice / 2-PR split the rebase plan above adopts: pull the engine out as its own PR (PR-A = R-1.6, providers.ts), then ship R-2 (PR-B) as four slices — contract, link (folds 2a's provisioning + 2c's heartbeat into one connection-lifecycle slice), backend (folds 2d's instant tile + 2e's metadata mirror into one slice), UX layer (2f as-is). Slice-by-slice scope and constraints live in R-2 rebase plan; this section just records that the retro's prescription is what's now being executed.

The original "one PR, atomic" judgment was load-bearing in two places (matched-pair invariants, no transitional master) but underweighted review-cycle cost. The 2-PR split addresses both: PR-A is a behavior-preserving refactor with no transitional master state (no remote tiles exist yet, half-wired or otherwise), and PR-B's matched-pair contract co-evolution still happens — just within PR-B's four commits, not across two PRs.

What the plan got right (and we'd keep)

  • Single TerminalBackend seam. R-1's interface absorbed remote with zero domain-knowledge bleed across the transport boundary. The router, surface, restore, and most of the UI never learned that "remote" exists. Vindicated by the 46-commit diff: ~85% of changed lines are in terminalBackend/remote.ts, agent.ts, and adjacent files — the rest of the codebase barely moved.
  • KOLU_AGENT_FLAKE_REF required, no fallback. Lesson #5 paid off — zero wire-shape-drift bugs in the prototype. Boot validation in httpServer.ts catches misconfigured deployments instantly with a clear error.
  • Deferred heartbeat via markConnected(). Lesson #6 paid off — multi-minute cold realisation on fresh hosts surfaced as "Connecting…" overlay, never as spurious disconnect. Captured here because the next remote-style integration (R-3, Kolu-on-NixOS, MCP adapter) will run into the same loops, and the human-in-the-loop cost of these mistakes was substantial — Codex caught 13 bugs across 4 review rounds, and the user had to interrupt 4 distinct premature-"done" stops to push the work forward.

Premature "done" on workflow-internal signals The agent declared the work complete four times on signals that weren't user-visible success — CI green ("End-to-end transport wired"); do-results step-end recorded ("R-2 ship complete"); a CI retry running in the background ("Standing by"); a just dev port bound + page reloaded ("ready for you to test"). Each required the user to interrupt: "wtf dont stop until you finish", "please finish the PR so it works end to end", "Did you finish everything?", "Why stop? Create a terminal and see what happens in logs". For UI/feature work, the turn ends only after the agent has driven the feature itself (chrome-devtools MCP, or direct CLI invocation) and observed the user-named outcome. If the only way for the user to confirm progress is to test it themselves, the turn is not over. CI-green and step-recorded are necessary, not sufficient.
Asymmetry blindness on local/remote variants Half of Codex round-1's catches (six bugs) were "the local backend publishes/disposes X; the remote backend doesn't." terminalExit subscription missing, RemotePtyHandle.dispose() never called, KOLU_AGENT_FLAKE_REF validation missing, repo-watch refcount leak, fs-watch refcount leak, kill-on-spawn-fail missing. The agent built the happy path and one error path, then stopped enumerating. When adding a symmetric variant of an existing concept, diff the new file against the existing one as an explicit checklist before declaring done. List every event the original publishes, every cleanup hook it runs, every refcount it manages — confirm each is mirrored. The asymmetry audit is cheap; finding the same gaps via review is expensive.
Cache invalidation needs an "every reader" sweep Codex round-2: HostSession.destroy() fired, but the destroyed instance still lived in two caches (one inside RemoteTerminalBackend, one inside getHostSession's pool). Fixing the first left the second returning the corpse. Round-3: a rejected sessionPromise cached without .catch-clearing, bricking the host until restart — while the sibling file remoteSession.ts already did this correctly. When invalidating cached state, grep for every reader of that cache before declaring the fix complete. And look for a sibling pattern that already does it right — copy from that, don't reinvent. "Fix at one site" is the bug.
Floating promises and sync callbacks must be traced Two crashes from "this looked safe so I didn't read the source": (a) void this.ensureStateSubscription() had no .catch, and on a missing-env path threw an unhandled rejection that killed the HTTP server; (b) waitForDisconnected called unsub() inside a synchronously-fired onState callback before const unsub = … had bound — TDZ ReferenceError on the already-disconnected branch. Whenever you write void someAsync(), open the primitive and confirm the rejection path is bounded. Whenever you attach a callback to an unknown subscription primitive, confirm whether it can fire synchronously on subscribe, and write the closure to tolerate that.
Harness-injected "Continue from where you left off" means work is unfinished Twice, a Bash command exited 137 (OOM kill of a heavy ssh probe), and Claude Code's harness re-prompted "Continue from where you left off." The agent interpreted this as "user has nothing more for me" and replied "No response requested" + ended the turn — requiring the user to manually re-nudge "Continue". Any harness-injected resume prompt means the prior turn ended mid-action. Treat it as "the work is unfinished — re-establish state from the last tool result and continue." Never respond to it as if it were the user opening a fresh conversation.
Cross-boundary feature additions need a "which side owns the filesystem?" check Codex round-3 caught: pasteImage / uploadFile wrote bytes to the parent's scratch dir then bracketed-pasted the path into the remote PTY — the path doesn't exist on the agent's host, so Claude/Codex running on the remote read garbage. The feature was added once and assumed location-agnostic. For any feature that writes a file then references it elsewhere, name explicitly which host owns the filesystem at each step. If the writer and reader can sit on different hosts, the feature needs either an explicit "remote not supported" guard (PRECONDITION_FAILED) or an agent-side write path — never a silent path that works locally and silently corrupts remotely.
API-name spelling: check the library's enum, not training-data convention Codex round-4: used FAILED_PRECONDITION (gRPC convention) for an ORPCError code. oRPC's defined codes only include PRECONDITION_FAILED; the unknown code fell through error-mapping to HTTP 500 instead of the intended 412 — silent dilution from "typed error" to "server crash." For library-defined enums (error codes, MIME types, signal names, HTTP statuses), open the library's source for the enum table rather than pattern-matching from neighbouring libraries. Differences in convention (gRPC vs oRPC, POSIX vs Windows, REST vs RPC) are real and silent.
