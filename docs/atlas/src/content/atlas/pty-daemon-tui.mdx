**A terminal-side client of kolu-server's pty-host.** kolu-server owns its PTYs
*in-process* today; `kolu-tui` connects to that same in-process pty-host over a local
unix socket (speaking `ptyHostSurface`) and gives you `list` / `attach` / `spawn`
from the terminal — no browser, **no daemon**. It is the CLI face of kolu and the
seed of a `tmux`/`zmx` replacement (<Issue n={671} />), shipped in phases as a beta.
*This doc is scoped to kolu-tui-on-the-in-process-server; the parent plan's
[Phase B](./pty-daemon.html#three-units-the-foundation-a1--a2-and-the-door-b)
(moving the pty-host into a surviving daemon) is a separate plan it later dovetails
with — see [Later](#later--how-this-dovetails-with-the-daemon-plan-out-of-scope).*

## Architecture — everything stays in-process in kolu-server

No new process. kolu-server keeps owning the PTYs exactly as it does now; it just
exposes the *same* in-process router over an additional local socket so a second
client — the CLI — can reach it. The web path is byte-identical.

<Arch>{`         ┌─────────────┐                          ┌──────────────────┐
         │   browser   │                          │     kolu-tui     │
         │  (the GUI)  │                          │ packages/pty-tui │
         └──────┬──────┘                          └─────────┬────────┘
                │  ws://  kolu-common/contract              │  ptyHostSurface
                │  rich: PTY + providers + sessions         │  raw: PTY + taps
                ▼                                           ▼
 ┌──────────────────────────────────────────────────────────────────────┐
 │  kolu-server   —   ONE process · all in-process · NO daemon            │
 │                                                                        │
 │                    servePtyHost(deps).router                          │
 │        (owns node-pty fds + @xterm/headless mirror + the VT taps)      │
 │                   ╱                              ╲                      │
 │            directLink                      serveOverStdio              │
 │       (web path — unchanged)            (NEW — additive, the SAME      │
 │              ╱                            router on a unix socket)     │
 │             ▼                                       ▼                   │
 │   LocalTerminalBackend                  ◆ $XDG_RUNTIME_DIR/kolu/       │
 │   (providers, sessions…)                  pty-host.sock ◆ ◀── kolu-tui │
 └──────────────────────────────────────────────────────────────────────┘`}</Arch>

**The split by richness.** The browser is the *rich* client — it speaks
kolu-server's public contract and gets the whole world (PTY plus the provider DAG:
git context, agent state, PR status, session grouping). `kolu-tui` is the *raw*
client — it speaks `ptyHostSurface` directly and gets the bare multiplexer (PTY plus
the VT taps), exactly the `tmux` altitude. Both attach to the one in-process
pty-host.

<Callout kind="note" title="The only server change is one additive socket">
`servePtyHost(deps).router` already exists and is already consumed in-process via
`directLink`. Phase 1 adds `serveOverStdio({ router })` on a unix socket beside it —
the same router, a second transport. The in-process web path doesn't change;
spawning, killing, and metadata all still flow through `LocalTerminalBackend` as
today. *No PTY moves, no process is added, nothing survives a restart that didn't
before.*
</Callout>

## Why a second client at all

Two reasons, both standing on their own without any reference to daemons.

- **It's a feature.** A persistent-while-the-server-runs terminal you drive from the shell — the CLI face of kolu, the start of the `tmux` replacement #671 has always pointed at.
- **It proves the contract is consumer-agnostic.** A1 (<PrLink pr={1055} />) asserts `ptyHostSurface` is a clean seam — kolu-server talks to the pty-host through a contract, not through `node-pty` internals. The only real proof of that is a *second, independent consumer*. If writing a terminal client against the surface is clean, the seam is at the right altitude (the "framework needs an end-to-end demo, not just unit tests" lesson — [parent plan, lesson #3](./remote-terminals.html)). If it's awkward, that's a finding worth having early.

<Callout kind="warn" title="Honest scope">
Because the pty-host is in-process, restarting kolu-server (a deploy) *kills* its
PTYs — `kolu-tui` does not make terminals survive a server restart. What it does
give is **client-side** detach/reattach: the CLI can come and go (close the terminal,
reopen, re-attach) while kolu-server keeps running and holds the PTY. Surviving a
*server* restart is the separate daemon plan
([B](./pty-daemon.html#three-units-the-foundation-a1--a2-and-the-door-b)), not this.
</Callout>

## The surface — ~150 LOC in `packages/pty-tui`

A small node CLI in its own package, `packages/pty-tui`, that dials the unix socket
through `stdioLink` (the same link the daemon would later use) and gets a typed
`ptyHostSurface` client. No new wire, no new framing — `serveOverStdio` is already
unit-tested via the loopback fixture.

| Subcommand | Surface call | Behaviour |
| --- | --- | --- |
| `kolu-tui list` | `terminal.list()` | One-shot — print each live PTY's id, pid, cwd, title. The pty-host's inventory. |
| `kolu-tui attach <id>` | `terminalAttach.get({id})` (stream) + `terminal.write` / `terminal.resize` | Put the local tty in raw mode. First yield is the **scrollback snapshot** → paint to stdout; deltas stream after. Forward stdin via `write`; `SIGWINCH` → `resize({id, cols, rows})`. A line-start `~.` escape (see [below](#decisions-recorded-here-to-be-echoed-in-code-comments)) detaches — the CLI exits, kolu-server keeps the PTY. |
| `kolu-tui spawn [-- cmd]` | `terminal.spawn({id: uuid, …})` then attach | Mint a UUID client-side, spawn, immediately attach. `--print-id` prints the id and does *not* attach (for scripting). |
| `kolu-tui snapshot <id>` | `terminalAttach.get({id})` — first yield only | One-shot, non-interactive: dump the current scrollback + a trailer line (`pid · cwd · title`) and exit. The scriptable primitive the [headless test](#b--headless-test--client-death-detachreattach-no-browser) asserts on. |
| `kolu-tui kill <id>` | `terminal.kill({id})` | Out-of-band termination — exercises the order-safe kill path (abort the exit tap *then* kill) from a non-browser client. |

The taps (`subscribeCwd` / `subscribeTitle` / `subscribeForeground` / `exitPromise`)
feed the status line — a second, non-browser consumer of the raw VT taps, proving
those are transport-clean too. `--json` makes `list` machine-readable; `--socket
<path>` points at a non-default server.

<Callout kind="note" title="Contract reality check">
The subcommand → surface mapping above is shorthand; these are the exact shapes the
*current* `ptyHostSurface` exposes (verified against `@kolu/pty-host` after A1
<PrLink pr={1055} /> and the link-family unification <PrLink pr={1059} />). Carry
these into the phases so the CLI is written against what the contract actually
offers, not the sketch:

- <Pill variant="warn">gap</Pill> **`terminal.list` returns `{ entries }`, and entries have no `title`.** The output is `{ entries: TerminalListEntry[] }` (not a bare array), and each entry is `{ id, pid, cwd, lastActivity }` — *title lives only on the title tap*. So Phase 1's `list` table must either drop the TITLE column or source it from `surface.title.get({id})` per row, and `list --json` / the headless test's `jq` must read `.entries[]`, not `.[]`.
- <Pill variant="warn">gap</Pill> **`terminal.spawn` has no command field — it always spawns `$SHELL`.** The spawn input is `{ id?, cwd?, cols?, rows?, scrollback? }`; there is no `command`/`argv`. So Phase 3's `spawn [-- cmd]` (and the headless test's `spawn --print-id -- bash -c …`) needs a deliberate decision: add a `command` field to the contract (a `PTY_HOST_CONTRACT_VERSION` bump) or write the command into the spawned shell. This is the one real contract gap behind the flow examples below.
- **The taps are host-method names; on the wire they're streams.** `subscribeCwd`/`subscribeTitle`/`subscribeForeground`/`exitPromise` are `PtyHost` methods below the seam — a CLI client consumes the surface *streams* `surface.cwd.get({id})` / `surface.title.get({id})` / `surface.foreground.get({id})` / `surface.exit.get({id})`. Note `exit` crosses as a one-shot *stream* (yields `{ exitCode }` once, then ends), not a `Promise` — the exit tap is awaited as the first frame of that stream.
- **The socket doesn't exist yet — and the path convention needs a decision.** No `serveOverStdio` listener exists in `packages/server` today (that *is* the Phase 1 work). But the doc's `$XDG_RUNTIME_DIR/kolu/pty-host.sock` diverges from the existing runtime-dir convention `$XDG_RUNTIME_DIR/kolu-<serverProcessId>/` (per-process, ephemeral — see `koluRoot.ts`). A discoverable socket arguably wants a *stable* `kolu/` dir, but Phase 1 should choose that path deliberately (and define the fallback when `XDG_RUNTIME_DIR` is unset) rather than inherit the per-PID dir by accident.

*What's unchanged and confirmed:* `servePtyHost(deps).router`, `directLink` (web
path), `serveOverStdio`, `stdioLink`, and the `terminalAttach.get({id})`
snapshot-then-delta stream all exist with the names this doc uses — the seam A1
promised is intact, and the shipped mini-ci example (Phase 0) already drives the same
`stdioLink` / `serveOverStdio` framing end-to-end.
</Callout>

### Decisions recorded here (to be echoed in code comments)

| Decision | Choice | Why |
| --- | --- | --- |
| **Home** | a tiny separate `packages/pty-tui` | An independent CLI package, not a `bin` folded into `@kolu/pty-host` — keeps the pty-host package focused on the contract + primitive, and keeps the CLI's own deps (raw-tty handling, arg parsing) out of the closure the A2 staleKey hashes. |
| **Render fidelity** | raw VT passthrough | Write the pty-host's bytes straight to stdout; do *not* re-render through a second `@xterm/headless` mirror. Simplest and truest — and it avoids a second rendering path drifting from the server's. *This trade-off must be recorded as a design-decision comment at the passthrough site in code.* |
| **Detach / escape** | ssh-style line-start `~` escape — **never Ctrl+B** | A passthrough multiplexer must not steal any control char the inner tools need — Ctrl+B and Ctrl+J are reserved by Claude Code (`input/prohibitedKeybinds.ts`), and editors/shells claim the rest. So the escape is the unambiguous ssh model: a `~` recognised *only* immediately after a newline you typed. `~.` detach · `~k` kill · `~~` literal tilde · `~?` help. The char is configurable (`--escape`); a literal `~` mid-line passes through untouched, so *every* inner-app chord, Ctrl+B included, reaches the program unmodified. |

## Phasing — three beta increments, all in-process

Phases 1–3 build kolu-tui on the in-process server — each leaves Kolu working, ships
on its own, and changes nothing in the web path. They're preceded by a **Phase 0**
that lives *outside* kolu entirely: a surface example that rehearses the whole
"interactive TUI over oRPC stdio" pattern
([detail below](#phase-0--a-minimal-ci-runner-tui-the-3rd-surface-example)).

| Phase | Ships | Server change | User-visible |
| --- | --- | --- | --- |
| **0 · surface example** <Pill variant="ok">shipped</Pill><br />(rehearsal, outside kolu) | A minimal CI-runner TUI over oRPC stdio in `@kolu/surface`'s examples — the 3rd example, after the worker demo + the remote-process-monitor (→ [drishti](https://github.com/srid/drishti)). Ships the runner the drishti way via `HostSession`; the default pipeline runs real typecheck CI. <Pill variant="ok">shipped</Pill> <PrLink pr={1073} /> | None — separate from kolu-server. | A standalone example (could graduate like drishti); proves the pattern and becomes kolu-tui's reference skeleton. |
| **1 · `list`**<br />(the socket + the round-trip) | kolu-server serves its in-process pty-host router over a unix socket (`serveOverStdio`); `packages/pty-tui` with `list` + read-only `snapshot`. | +1 local socket listener on the existing router. Web path byte-identical. | A beta CLI that *lists* your live terminals. + a "Terminal UI (beta)" line in the README/site. |
| **2 · `attach`**<br />(the interactive client) | Raw-tty passthrough, snapshot-then-delta, stdin→`write`, `SIGWINCH`→`resize`, the `~`-escape detach. | None beyond Phase 1. | Drive a terminal from the CLI; detach and re-attach while the server runs. |
| **3 · `spawn` / `kill`**<br />(the raw multiplexer) | `spawn` (+ `--print-id`), `kill`, `list --json`. | None. | Create and kill PTYs from the CLI — a usable raw multiplexer (beta). The richer `tmux`-replacement UX (windows, panes, config) grows from here, as its own work. |

Phase 1 alone is the smallest honest first ship of *kolu-tui* — exactly the `kolu-tui
list` starting point: a one-shot RPC, no raw-tty mode, the "hello world" that proves
`serveOverStdio`-over-a-real-socket + `stdioLink` + the contract round-trip, at
near-zero risk.

### Phase 0 — a minimal CI-runner TUI (the 3rd surface example) <Pill variant="ok">shipped</Pill> <PrLink pr={1073} />

Before touching kolu at all, prove the pattern in `@kolu/surface`'s examples — the
falsifiability test of [lesson #3](./remote-terminals.html) applied to "interactive
TUI over oRPC stdio," the way the worker demo and the remote-process-monitor (which
became [drishti](https://github.com/srid/drishti)) validated the earlier patterns.
The candidate: a **minimal CI runner** — justci-*flavoured* but self-contained,
deliberately *not* the real [justci](https://github.com/juspay/justci) (no Haskell,
no GitHub statuses, no multi-platform fan-out). Just a small DAG of shell commands —
runnable locally, or on a single remote host where the prebuilt runner closure is
shipped the [drishti](https://github.com/srid/drishti) way (`nix copy` + realise over
ssh, via `@kolu/surface-nix-host`'s `HostSession`; see
[below](#remote-mode--the-drishti-hostsession-closure-ship-over-ssh)).

<Callout kind="note" title="As built (#1073, merged)">
Shipped as `@kolu/surface-example-mini-ci`. Two things landed differently from this
section's original sketch, both deliberately: the remote transport *pivoted from
`git archive` source-ship to the drishti `HostSession` closure-ship* (so the example
exercises the very R-2 transport the parent plan builds on, not a minimal cousin of
it), and the default pipeline *runs real typecheck CI for the remote-process-monitor
example* (a `surface` + `nix-host` → `monitor` diamond of `tsc --noEmit` gates)
rather than a toy `build → test → lint`. The dashboard also binds **plain keys**
(digits/`n`/`p`/`r`/`q`), *not* the `~`-escape — it renders structured state and owns
the keyboard, so it has no need for kolu-tui's raw-passthrough escape (that decision
is recorded for kolu-tui below). One constraint worth carrying forward: the runner
ships as a read-only nix closure, so only read-only tasks (typecheck) belong in the
default pipeline — write-heavy steps (`vite build`, `nix build`) would need a writable
copy of the workspace first.
</Callout>

It's a clean structural twin of kolu-tui: a long-lived runner owns the DAG + each
node's process + its log buffer, and streams to ephemeral TUI clients over stdio. The
plan's `nodes.list()` / `node.log(id)` / `node.rerun(id)` map onto the
framework-idiomatic spelling the surface derives — a `nodes` *cell*
(`surface.nodes.get({})`) ↔ kolu-tui's `list`; a `nodeLog` *stream*
(`surface.nodeLog.get({id})`, snapshot-then-delta) ↔ `attach`; a `node.rerun`
*procedure* (`surface.node.rerun({id})`) ↔ input. That this was clean to write against
the surface primitives is the finding: the seam is at the right altitude for kolu-tui
to inherit. The example is now kolu-tui's copy-paste skeleton (raw-tty setup, the
snapshot-paint loop) and a permanent regression test for the pattern.

#### Remote mode — the drishti `HostSession` closure-ship over ssh <Pill variant="ok">as-built</Pill>

mini-ci runs the DAG locally by default, but it can also run on a **remote host** —
and it ships the runner the [drishti](https://github.com/srid/drishti) way, exactly
as the remote-process-monitor example does: a prebuilt `mini-ci-runner` nix closure
is shipped with `nix copy` to the host, realised with `nix-store --realise`, then run
as `ssh host mini-ci-runner --stdio`, with the TUI attached over **stdio-over-ssh**.
`connect()` reads the host-arch runner `.drv` from `MINI_CI_RUNNER_DRV` (required) and
hands it to `@kolu/surface-nix-host`'s `getHostSession({ host, binary, resolveDrvPath })`,
which owns the ref-count, reconnect, watchdog, and a `copying → connecting → connected`
state cell. `localhost` skips the `nix copy` and runs the realised binary directly —
the *same* `HostSession`, only the transport differs. So the example exercises the
surface's stdio link in *both* placements over the one `HostSession` — the same R-2 /
drishti closure-ship transport the parent plan builds on. *(This is the drishti
transport itself, not a `git archive` cousin of it: a closure, not source — the
trade-off being that only read-only tasks belong in the shipped closure.)*

  └──────────────┘                                    │   2. run the DAG over ssh stdio │

### B · Headless test — client-death detach/reattach, no browser

The detach/reattach loop scripted. It asserts that the CLI client can die and a fresh
one re-joins the same PTY (the server runs throughout) — fast, no browser.

```bash
#!/usr/bin/env bash
set -euo pipefail

# 1 · spawn a PTY with a unique marker in its scrollback (no attach)
id=$(kolu-tui spawn --print-id -- bash -c 'echo "MARK-$$"; exec sleep 1d')
pid_before=$(kolu-tui list --json | jq -r ".[] | select(.id==\"$id\").pid")

# 2 · (no kolu-tui process is attached — simulating client death)

# 3 · a fresh client re-attaches by id and asserts the state is intact
kolu-tui snapshot "$id" | grep -q "MARK-"            # scrollback preserved
pid_after=$(kolu-tui list --json | jq -r ".[] | select(.id==\"$id\").pid")
[ "$pid_before" = "$pid_after" ]                     # same PTY, not a respawn

echo "✓ client came and went; server held the PTY (pid $pid_after unchanged)"
```

This is the kolu-tui-scoped reattach test — the client can come and go while the
server holds the PTY. (Surviving a *server* restart is a different assertion that
belongs to the daemon plan, not here.)

## Later — how this dovetails with the daemon plan (out of scope)

Stated only so we don't trip over it: when the parent plan's
[Phase B](./pty-daemon.html#three-units-the-foundation-a1--a2-and-the-door-b)
eventually moves the pty-host out of kolu-server into a surviving `kolu --stdio`
daemon, `kolu-tui`'s socket target shifts from kolu-server to the daemon — **with no
contract change**, because both serve the same `ptyHostSurface`. At that point the
CLI's terminals gain server-restart survival for free, and the raw-vs-rich client
split is unchanged. *But that is a separate plan; nothing in kolu-tui's phases above
depends on it, and kolu-tui ships and is useful entirely on the in-process server.*
